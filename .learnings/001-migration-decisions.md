# Migration Decisions: Figma React/Vite → Next.js

## Framework Choice: Next.js over Astro
- React-based so existing components port directly without rewriting
- Best Sanity integration via `next-sanity` package
- App Router enables server components for data fetching, client components for interactivity

## Dead Code Discovery
- All 58 shadcn/ui components generated by Figma were unused (zero imports from app code)
- The `src/imports/Body.tsx` Figma export was also dead code
- Dropped all of them, keeping only `cn()` utility from shadcn

## Atomic Design Classification
- **Atoms**: Single-purpose elements (BlobLogo, HorizontalLine, TypeBadge, NavLink, TimeDisplay)
- **Molecules**: Small composed groups (RevealText, FadeIn, StaggerGroup, CommentCard, PostCard, TimelineEntry)
- **Organisms**: Large interactive sections (BlobCursor, Navbar, Footer, PostList, TimelineFeed, CommentSection, HeroSection)
- **Templates**: Page-level wrappers (PageShell for AnimatePresence transitions)

## Client vs Server Component Boundary
- Pages are server components (data fetching happens at the page level)
- Interactive organisms/molecules are client components (`'use client'`)
- BlobCursor uses `next/dynamic` with `ssr: false` (canvas + window access)

## Sanity Fallback Pattern
- `src/lib/sanity/fetch.ts` checks if `NEXT_PUBLIC_SANITY_PROJECT_ID` is set
- If not, returns data from `src/lib/fallback-data.ts` (hardcoded posts/timeline)
- This allows the site to work immediately without Sanity configuration

## Dependencies Audit
- Removed ~40 packages (all @radix-ui, @emotion, @mui, react-router, etc.)
- Kept: react, motion, lucide-react, date-fns, clsx, tailwind-merge, next-themes
- Added: next, sanity, next-sanity, @sanity/client, @portabletext/react
